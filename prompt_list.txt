## 项目开发历程总结

### 第一阶段：问题诊断
- 发现 Ed25519 密钥解析失败（keyType 为空）
- 定位到 `numKeys` 解析错误（应为 uint32，实际只读了 1 字节）

### 第二阶段：Ed25519 支持
- 添加 TweetNaCl.js 库支持 Ed25519 签名
- 实现 Ed25519 密钥解析和签名

### 第三阶段：MAC 检查失败
- 发现序列号竞争条件问题
- `handlePacket` 可能在 `incomingSeq++` 之前触发新的消息处理
- 修复：在调用 `handlePacket` 之前先递增 `incomingSeq`

### 第四阶段：签名数据错误
- 发现 Session ID 被错误序列化为逗号分隔字符串
- 原因：`appendString` 对 Uint8Array 调用 `TextEncoder.encode()` 会先转为字符串
- 修复：使用 `appendBuffer` 代替 `appendString`

- 添加英文 README.md
- 添加中文 README_CN.md

## 关键 Bug 修复

### Bug 1: Ed25519 密钥解析失败
**问题**：`Unsupported key type: ` (空字符串)
**原因**：`numKeys` 读取错误，导致后续偏移量错位
**修复**：
```javascript
// 错误
const numKeys = raw[offset++];
offset++;

// 正确
const numKeys = (raw[offset] << 24) | (raw[offset+1] << 16) | 
                (raw[offset+2] << 8) | raw[offset+3];
offset += 4;
```

### Bug 2: MAC 检查失败（序列号竞争）
**问题**：收到签名响应时 MAC 检查失败，序列号错误
**原因**：`handlePacket` 发送数据包后，服务器立即响应，但 `incomingSeq++` 还未执行
**修复**：
```javascript
// 错误
await this.handlePacket(payload, decrypted);
this.incomingSeq++;

// 正确
this.incomingSeq++;
await this.handlePacket(payload, decrypted);
```

### Bug 3: Ed25519 签名验证失败
**问题**：服务器返回 USERAUTH_FAILURE
**原因**：Session ID 被序列化为 `"203,5,138,53,..."` 而不是原始字节
**修复**：
```javascript
// 错误
sessionData.appendString(this.sessionId);

// 正确
sessionData.appendBuffer(this.sessionId);
```
